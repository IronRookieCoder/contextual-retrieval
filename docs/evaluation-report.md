## 三、评估指标

### 3.1 Pass@k

在 top-k 检索结果中，查找是否包含目标块（精确内容匹配）。Pass@k = 被找到的目标块数 ÷ 总目标块数。

### 3.2 Precision@k

Precision@k = 前 k 个结果中相关的块数 ÷ k。

### 3.3 Recall@k

Recall@k = 前 k 个结果中相关的块数 ÷ 总相关块数（即目标块数）。

### 3.4 MRR (Mean Reciprocal Rank)

MRR = 第一个相关块排名的倒数。衡量系统将最相关结果排到前列的能力。

## 四、Bug 修复记录

在评估过程中发现了源码中的若干 Bug 并进行了修复：

### 4.1 评估器字段名不匹配

**问题**: `evaluation.py` 的 `extract_golden_contents()` 方法中：
- 查找 Golden 文档时用 `doc.get("uuid")`，但数据集使用 `original_uuid`
- 查找 Golden 块时用 `chunk.get("index")`，但数据集使用 `original_index`
- 提取检索结果内容时用 `original_content`，但 Base RAG 存储的是 `content`

**修复**: 统一兼容两种字段名，并提取公共方法 `_extract_content()`。

**影响**: 修复前所有评估结果均为 0%。

### 4.2 BM25 批量索引返回格式

**问题**: `bm25_search.py` 中 `bulk()` 函数在 elasticsearch 9.x 返回 `(success_count, error_list)`，但代码将其当作 `(success_count, error_count)`，导致 `failed > 0` 类型错误。

**修复**: `failed = len(errors) if isinstance(errors, list) else errors`。

### 4.3 混合搜索引擎字段映射

**问题**: `hybrid_search.py` 中 RRF 融合时，语义结果 metadata 使用 `original_content`，BM25 结果使用 `content`，字段名不统一导致 KeyError。

**修复**: 统一映射为 `original_content` 输出。

### 4.4 数据生成器字段名

**问题**: `data_generator.py` 需要同时支持 `uuid` 和 `original_uuid`、`index` 和 `original_index`。

**修复**: 同时写入两个字段。

## 八、结论与建议

### 8.1 核心结论

1. **上下文增强 RAG 在真实文档上一致优于基础 RAG**
   - Pass@10: **+5.2%**（69.2% → 74.4%）
   - Pass@20: **+12.8%**（76.9% → 89.7%）
   - MRR: **+14.4%**（0.4581 → 0.5240）

2. **一次性成本极低**
   - 262 个真实文档块的上下文生成仅需 ~$0.34
   - 对 1 万篇文档规模预估 ~$260，仍然经济可行

3. **基础 RAG 在真实文档上表现尚可**
   - Jina Embeddings v4 对真实中文技术文档的语义理解较好
   - 模板化测试数据会显著低估基础 RAG 的真实能力

4. **混合搜索（+BM25）在真实文档上提升有限**
   - 与合成数据结果不同，真实文档中混合搜索未显著超越纯上下文搜索
   - Pass@5/10 持平，Pass@20 略降 2.5%（BM25 的关键词匹配在大 k 值下引入噪声）
   - MRR 略低 2.0%（0.5240 → 0.5138），说明混合搜索未能将目标结果排到更前的位置
   - 原因分析：Jina Embeddings v4 对真实技术文档的语义理解已足够好，BM25 的补充收益被噪声抵消

### 8.2 使用建议

| 场景 | 推荐方案 |
|------|----------|
| 小型知识库 (< 100 篇文档) | 上下文增强 RAG |
| 中型知识库 (100-10,000 篇) | 上下文增强 RAG + 混合搜索 |
| 大型知识库 (> 10,000 篇) | 上下文增强 RAG + 混合搜索 + 重排序 |
| 对精度要求极高 | 过检索 + Jina Reranker v3 精排 |

### 8.3 进一步优化方向

- **更好的块切分策略**: 语义分块（Semantic Chunking）而非固定大小/标题分块
- **更细粒度的上下文**: 对于超长文档，可引入父子块结构（父块提供上下文，子块作为检索单元）
- **多轮优化**: 对 DeepSeek 生成的上下文进行质量评估和迭代优化
- **增量更新**: 支持增量添加文档而不需重建全量索引
